# Story 31.1: Extend EVMClaimMessage with Self-Describing Fields

**Epic:** 31 - Self-Describing BTP Claims & Dynamic Channel Verification
**Story Number:** 31.1
**Status:** Done

## Story

**As a** connector developer,
**I want** BTP EVM claims to carry chain and contract coordinates (`chainId`, `tokenNetworkAddress`, `tokenAddress`),
**so that** claims are self-describing and a receiving connector can verify unknown channels on-chain without requiring pre-registration via the Admin API.

## Acceptance Criteria

1. `EVMClaimMessage` interface includes `chainId?`, `tokenNetworkAddress?`, `tokenAddress?` (optional for backward compatibility)
2. `validateEVMClaim()` validates new fields when present (correct types, 0x-prefix for addresses)
3. `validateEVMClaim()` passes when new fields are absent (backward compatibility)
4. `ClaimSender.sendEVMClaim()` populates new fields from existing channel metadata and settlement config
5. Unit tests cover validation with/without new fields, correct/incorrect formats

### Data Models

**Current `EVMClaimMessage` interface** (`packages/connector/src/btp/btp-claim-types.ts:68-77`):

```typescript
export interface EVMClaimMessage extends BaseClaimMessage {
  blockchain: 'evm';
  channelId: string;
  nonce: number;
  transferredAmount: string;
  lockedAmount: string;
  locksRoot: string;
  signature: string;
  signerAddress: string;
}
```

**New fields to add** (all optional for backward compatibility with pre-registered channels):

- `chainId?: number` -- EVM chain ID (e.g., 8453 for Base, 84532 for Base Sepolia, 31337 for Anvil)
- `tokenNetworkAddress?: string` -- TokenNetwork contract address (0x-prefixed, 40 hex chars)
- `tokenAddress?: string` -- ERC20 token contract address (0x-prefixed, 40 hex chars)

These fields are already part of the EIP-712 signing domain (`{ name: 'TokenNetwork', version: '1', chainId, verifyingContract: tokenNetworkAddress }`), so they are cryptographically bound to the signature -- tampering causes signature verification to fail. [Source: docs/architecture/data-models.md#EVMClaimMessage, docs/connector-self-describing-claims.md#Change-1]

**`ChannelMetadata`** (`packages/connector/src/settlement/channel-manager.ts:33-42`) already stores:

- `tokenAddress: string` -- ERC20 token contract address
- `chain: string` -- Chain identifier (e.g., "evm:base:8453")

**`ChannelManagerConfig`** (`packages/connector/src/settlement/channel-manager.ts:16-28`) has:

- `registryAddress: string` -- TokenNetworkRegistry contract address
- `rpcUrl: string` -- Base L2 RPC URL

### Sending Side: Where Values Come From

The `sendEVMClaim()` method is called from `UnifiedSettlementExecutor.settleViaEVM()` (`packages/connector/src/settlement/unified-settlement-executor.ts:296`). At the call site, the executor already has:

- `tokenAddress` -- passed as a parameter to `settleViaEVM()`
- `channelId` -- obtained from `openChannel()` result

The `PaymentChannelSDK` (`packages/connector/src/settlement/payment-channel-sdk.ts`) internally uses these values during `signBalanceProof()` but does not expose them publicly:

- `chainId` -- via `this.provider.getNetwork().chainId` (line 521-522, private `provider`)
- `tokenNetworkAddress` -- via `contract.getAddress()` from `tokenNetworkCache` (line 502-507, private cache)

**To expose these values**, add two public accessor methods to `PaymentChannelSDK`:

```typescript
/** Get the chain ID from the connected provider */
async getChainId(): Promise<number> {
  const network = await this.provider.getNetwork();
  return Number(network.chainId);
}

/** Get the TokenNetwork contract address for a given token */
async getTokenNetworkAddress(tokenAddress: string): Promise<string> {
  const tokenNetwork = await this.getTokenNetworkContract(tokenAddress);
  return await tokenNetwork.getAddress();
}
```

Note: `getTokenNetworkContract()` is an existing private method (line 201) that resolves and caches the TokenNetwork contract. Both methods reuse existing internal logic with no new external calls.

**Approach:** Extend `sendEVMClaim()` to accept three new optional parameters: `chainId`, `tokenNetworkAddress`, `tokenAddress`. The caller (`UnifiedSettlementExecutor`) will obtain `chainId` and `tokenNetworkAddress` via the new `PaymentChannelSDK` accessor methods, and pass all three values along with the existing `tokenAddress` parameter. The fields are optional in the method signature so callers without these values (backward compat) can omit them.

### EIP-712 Domain Separator

The `getDomainSeparator()` helper (`packages/connector/src/settlement/eip712-helper.ts:14-29`) constructs:

```typescript
{ name: 'TokenNetwork', version: '1', chainId, verifyingContract: tokenNetworkAddress }
```

This means `chainId` and `tokenNetworkAddress` are already cryptographically bound to every EIP-712 signature. Story 31.2 will use the claim's self-describing fields to reconstruct this domain for verification of unknown channels.

### Security Review

**No security concerns.** The new fields are cryptographically bound to the EIP-712 signature via the domain separator (`chainId` and `tokenNetworkAddress` are domain parameters). Tampering with these fields causes signature verification to fail in the receiving connector (Story 31.2). This is correctly documented in the JSDoc and PRD.

The validation regex `/^0x[0-9a-fA-F]{40}$/` is robust -- it prevents injection of non-hex characters and enforces exact length. The `chainId` validation (`Number.isInteger(chainId) && chainId > 0`) correctly rejects zero, negative, fractional, string, NaN, and Infinity values.

### Performance Considerations

No performance concerns. The new fields add approximately 120 bytes per serialized claim message. The two new async calls in `settleViaEVM` (`getChainId()` and `getTokenNetworkAddress()`) use cached values internally (provider network is cached by ethers.js; TokenNetwork contract is cached in `tokenNetworkCache`), so latency impact is negligible.
