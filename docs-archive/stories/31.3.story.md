# Story 31.3: Integration Tests & Backward Compatibility Verification

**Epic:** 31 - Self-Describing BTP Claims & Dynamic Channel Verification
**Story Number:** 31.3
**Status:** Done

## Story

**As a** connector developer,
**I want** end-to-end integration tests that validate the complete self-describing claims flow against a local Anvil blockchain and verify backward compatibility with pre-registered channels,
**so that** I have confidence the dynamic on-chain verification works correctly in a realistic multi-component environment and existing functionality is not broken.

## Acceptance Criteria

1. First contact flow test passes: unknown peer connects via BTP, sends self-describing claim, connector verifies on-chain, registers channel and peer
2. Caching test passes: subsequent claims from the same channel skip RPC (verify no additional RPC calls)
3. Tampered fields test passes: claim with modified `chainId` or `tokenNetworkAddress` fails EIP-712 signature verification
4. Backward compat test passes: pre-registered channel (via Admin API / `ChannelManager.ensureChannelExists()`) continues to work with claims that omit new fields
5. Mixed mode test passes: pre-registered channel works with claims that include new fields (fields are informational, channel already cached)
6. All 5 test scenarios pass with no regression in existing claim exchange tests
7. Tests run in CI without external dependencies beyond Anvil (no external RPC, no Mainnet)

### EIP-712 Domain Construction

The EIP-712 domain for claims is `{ name: 'TokenNetwork', version: '1', chainId, verifyingContract: tokenNetworkAddress }`. For self-describing claims, the domain is reconstructed from the claim's own fields. Tampered fields produce different errors depending on which field is modified: [Source: packages/connector/src/settlement/eip712-helper.ts, claim-receiver.ts:250-334]

- **Tampered `chainId`:** The on-chain query (`getChannelStateByNetwork`) does NOT use `chainId`, so the channel is found and passes state/participant checks. However, `verifyBalanceProofWithDomain()` reconstructs the EIP-712 domain with the wrong `chainId`, causing signature recovery to produce a different address. Error returned: `'Invalid EIP-712 signature'` (literal string, not an `ERRORS` constant). [Source: claim-receiver.ts:329-334]
- **Tampered `tokenNetworkAddress`:** The on-chain query targets the wrong contract address. This either throws (error: `ERRORS.ON_CHAIN_VERIFICATION_FAILED`) or returns a non-existent channel (error: `ERRORS.CHANNEL_NOT_FOUND`). [Source: claim-receiver.ts:251-264, 268-275]

### On-Chain Channel State

The TokenNetwork contract `channels(channelId)` returns `(settlementTimeout, state, closedAt, openedAt, participant1, participant2)`. State enum: 0=NonExistent, 1=Opened, 2=Closed, 3=Settled. [Source: packages/connector/src/settlement/payment-channel-sdk.ts]
