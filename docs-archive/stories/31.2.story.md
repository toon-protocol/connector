# Story 31.2: Dynamic On-Chain Channel Verification & Auto-Peer Registration

**Epic:** 31 - Self-Describing BTP Claims & Dynamic Channel Verification
**Story Number:** 31.2
**Status:** Done

## Story

**As a** connector operator,
**I want** the connector to verify unknown channels on-chain using self-describing claim fields and auto-register peers on first contact,
**so that** peers can open payment channels unilaterally and begin transacting without Admin API pre-registration or SPSP handshake.

## Acceptance Criteria

1. Claims for unknown channels with valid on-chain state are accepted
2. Claims for non-existent or closed channels are rejected with clear error
3. Claims where `signerAddress` is not a channel participant are rejected
4. Verified channels are cached -- second claim skips RPC
5. Peer is auto-registered and routable after first verified claim
6. Unit tests with mocked RPC responses cover all verification branches

### Component Specifications

**ClaimReceiver** (`packages/connector/src/settlement/claim-receiver.ts`):

- Current `verifyEVMClaim()` (line 184-234) verifies EIP-712 signature via `evmChannelSDK.verifyBalanceProof()` and checks nonce monotonicity against the latest verified claim in the DB
- Current flow assumes the channel is already known (pre-registered). For unknown channels, `verifyBalanceProof()` iterates `tokenNetworkCache` to find the contract -- this fails for channels on unknown TokenNetworks
- **New behavior**: Before signature verification, check if the channel exists in `ChannelManager.channelMetadata`. If NOT:
  1. Require all three self-describing fields (`chainId`, `tokenNetworkAddress`, `tokenAddress`) -- reject if any absent. `tokenAddress` is required because `ChannelMetadata.tokenAddress` is non-optional
  2. Query on-chain state via `PaymentChannelSDK.getChannelState()` using the claim's `tokenNetworkAddress` (NOT via registry lookup -- use contract address directly)
  3. Verify channel exists and `status === 'opened'`
  4. Verify `claim.signerAddress` matches `participant1` or `participant2` on-chain
  5. Verify EIP-712 signature using domain from claim fields: `getDomainSeparator(claim.chainId, claim.tokenNetworkAddress)`
  6. Register channel in `ChannelManager` via new `registerExternalChannel()` method
  7. Process normally
- **Constructor change**: `ClaimReceiver` needs access to `ChannelManager` to check and register channels. Add `channelManager` as a new constructor parameter (optional for backward compat with existing tests).
- **Signature verification for unknown channels**: The current `verifyBalanceProof()` on `PaymentChannelSDK` relies on iterating `tokenNetworkCache` which only contains contracts registered through the registry. For unknown channels, we need a method that accepts explicit `chainId` and `tokenNetworkAddress` to construct the EIP-712 domain directly.

[Source: docs/architecture/components.md#ClaimReceiver, docs/architecture/core-workflows.md#Dynamic-On-Chain-Verification-Workflow]

**ChannelManager** (`packages/connector/src/settlement/channel-manager.ts`):

- Current methods: `getChannelById()` (line 164), `getChannelForPeer()` (line 153), `ensureChannelExists()` (line 127)
- Internally stores: `channelMetadata: Map<string, ChannelMetadata>` (channelId → metadata) and `peerChannelIndex: Map<string, Map<string, string>>` (peerId → tokenId → channelId)
- **New method needed**: `registerExternalChannel(params)` to register a channel discovered from an incoming self-describing claim. This populates both `channelMetadata` and `peerChannelIndex` without opening a channel on-chain.
- **Token ID resolution**: For externally discovered channels, we don't have a `tokenId`. Use a derived token ID from the token address (e.g., reverse-lookup from `config.tokenAddressMap`, or use the address itself as a fallback token ID).

### EIP-712 Domain Construction for Unknown Channels

The EIP-712 helper (`packages/connector/src/settlement/eip712-helper.ts:14-29`) provides:

```typescript
function getDomainSeparator(chainId: number | bigint, verifyingContract: string) {
  return { name: 'TokenNetwork', version: '1', chainId, verifyingContract };
}
```

For unknown channels, reconstruct the domain from the self-describing claim fields:

```typescript
const domain = getDomainSeparator(claim.chainId, claim.tokenNetworkAddress);
```

This ensures cryptographic binding: if `chainId` or `tokenNetworkAddress` in the claim are tampered with, the recovered signer address will differ from the expected signer, causing verification to fail. [Source: docs/connector-self-describing-claims.md#Change-2, packages/connector/src/settlement/eip712-helper.ts]

### Verification Flow (First Contact)

Per the architecture workflow diagram [Source: docs/architecture/core-workflows.md#Dynamic-On-Chain-Verification-Workflow]:

1. ClaimReceiver receives claim with `channelId` NOT in `channelMetadata`
2. Extract contract coordinates: `chainId`, `tokenNetworkAddress`, `channelId` from claim
3. RPC call: `channels(channelId)` on `tokenNetworkAddress` at `chainId`
4. Verify: channel exists, state === opened (1), `signerAddress` matches `participant1` or `participant2`
5. Verify EIP-712 signature with domain: `{ name: 'TokenNetwork', version: '1', chainId: claim.chainId, verifyingContract: claim.tokenNetworkAddress }`
6. Register channel in `ChannelManager` (add to `channelMetadata` and `peerChannelIndex`)
7. Auto-register peer: the peer is already connected via the BTP WebSocket that delivered the claim. Registering the channel in `ChannelManager` (adding to `channelMetadata` and `peerChannelIndex`) is sufficient for the peer to be "routable" for settlement purposes. No separate routing table update is needed -- the BTP connection already provides ILP packet delivery, and `ChannelManager` registration enables settlement claim processing.
8. Process normally -- subsequent claims skip RPC (fast path)

### On-Chain Channel State Mapping

The TokenNetwork contract `channels(channelId)` returns:

```
(uint256 settlementTimeout, uint8 state, uint256 closedAt, uint256 openedAt, address participant1, address participant2)
```

State enum mapping:

- `0` = NonExistent
- `1` = Opened
- `2` = Closed
- `3` = Settled

For verification, we require `state === 1` (Opened). [Source: packages/connector/src/settlement/payment-channel-sdk.ts:800-806]
